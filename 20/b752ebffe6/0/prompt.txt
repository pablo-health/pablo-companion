## Unit Test Analysis: Pablo Companion (macOS)

### What's Worth Testing (Prioritized)

**1. `AuthViewModel.handleAuthCallback(callbackURL:error:)` ã HIGH VALUE, MEDIUM EFFORT**

Why it earns its keep: This is the most complex pure-logic function in the app. It parses a callback URL, extracts query parameters, validates required tokens, stores them in Keychain, extracts email from JWT, and sets auth state. A bug here means users can't sign in or lose their session. The function has 5+ distinct code paths (cancelled login, error, missing callback, missing tokens, happy path). This is exactly the kind of branching logic where real bugs hide.

Real bugs it catches:
- Missing `id_token` or `refresh_token` silently fails instead of showing error
- JWT email extraction returns nil and falls through incorrectly
- Auth state gets stuck in `.authenticating` if error path is incomplete

**2. `AuthViewModel.decodeJWTPayload(_:)` and `extractExpiry(from:)` ã HIGH VALUE, LOW EFFORT**

Why it earns its keep: Base64 URL-safe decoding with padding is notoriously fiddly. The function handles `-` to `+`, `_` to `/` replacements and padding to multiples of 4. This is pure logic, no dependencies, trivially testable.

Real bugs it catches:
- Tokens with unusual padding lengths decode incorrectly
- JWT with non-standard characters fail silently
- Expiry extraction returns nil for valid tokens

**3. `AuthViewModel.getValidToken()` ã token expiry/refresh logic ã HIGH VALUE, MEDIUM EFFORT**

Why it earns its keep: The 5-minute buffer calculation determines whether a token refresh happens. Off-by-one in the time comparison means either unnecessary refreshes (perf hit) or expired tokens hitting the API (auth failures).

Real bugs it catches:
- Token refresh triggered too early or too late
- Missing token returns wrong error type
- State transitions (`.tokenExpired`) happen at wrong time

**4. `DemoEncryptor.decryptFile(at:)` ã MEDIUM VALUE, LOW EFFORT**

Why it earns its keep: Binary format parsing with chunk length prefixes and WAV header reconstruction. The offset arithmetic (`offset + 4`, little-endian UInt32) is exactly the kind of code that breaks silently with corrupt data. Real bugs it catches:
- Off-by-one in chunk boundary parsing
- WAV header size fields incorrect after reconstruction
- Truncated file crashes instead of throwing

**5. `APIClient.createMultipartBody(fileData:fileName:boundary:)` ã MEDIUM VALUE, LOW EFFORT**

Why it earns its keep: Multipart form data construction must have exact `\r\n` placement and boundary formatting. If the boundary or content disposition is wrong, uploads silently fail server-side.

Real bugs it catches:
- Missing `\r\n` terminators
- Boundary not matching Content-Type header format
- Special characters in filename breaking the form data

**6. `RecordingViewModel.checkBluetoothRoutingConflict()` ã MEDIUM VALUE, LOW EFFORT**

Real bugs it catches:
- UIDs without ":" separator cause crash or wrong comparison
- Prefix match too loose (unrelated devices matching)

**7. `LocalRecording.formattedDuration` ã LOW VALUE, LOW EFFORT**

Only worth testing as a freebie since it's pure logic with no dependencies. Could catch edge cases like durations over 59:59 not displaying hours.

## CodeQL Slowness Investigation: Complete Findings

### 1. What Specifically Causes CodeQL to Hang on Package Checking

The root cause is **SPM package resolution running from scratch on every CI run**, compounded by two specific issues in this project:

**A. AudioCaptureKit is pinned to `branch = main` (not a version tag)**

In `mac/PabloCompanion.xcodeproj/project.pbxproj` (lines 351-358):
```
XCRemoteSwiftPackageReference "AudioCaptureKit" = {
    repositoryURL = "https://github.com/lll-solutions/AudioCaptureKit.git";
    requirement = {
        branch = main;
        kind = branch;
    };
};
```

Branch-based pins are the worst case for CI because SPM must **check the remote every single build** to see if `main` has new commits. Version-pinned packages can skip the network check if the resolved version matches. The `Package.resolved` file records a specific revision (`e378e950ca05cf8c6dc56167fb34b21ebf34e1f7`), but SPM still needs to verify `main` hasn't moved.

**B. SPM fetches full git history by default.** When SPM resolves, it clones the full repository (all branches, full history). AudioCaptureKit pulls in transitive dependencies: `swift-crypto` and `swift-asn1` (both Apple packages). Apple packages are known to be especially slow to clone due to large git histories.

**C. No SPM caching in any workflow.** The CodeQL workflow (`codeql.yml`) does a bare `xcodebuild build` with no caching step. Every run does a full fresh SPM resolution + clone. The CI workflow (`ci.yml`) has the same problem.

**D. Known GitHub Actions issue.** There's a documented issue (actions/runner-images#6233) where SPM hangs on xcodebuild in GitHub Actions due to `.netrc`/Keychain interaction, particularly for HTTPS-based packages on macOS runners.

### 2. Does the Current Workflow Already Cache SPM Packages?

**No.** Neither `codeql.yml` nor `ci.yml` has any `actions/cache` step for SPM. The Rust jobs (`ci.yml` lines 106, 126) correctly use `Swatinem/rust-cache@v2`, but the Swift jobs have zero caching.

### 3. Is AudioCaptureKit Local or Remote?

**Remote.** It's an `XCRemoteSwiftPackageReference` pointing to `https://github.com/lll-solutions/AudioCaptureKit.git` with `kind = branch` pinned to `main`.

The `Package.resolved` file confirms three remote dependencies:
- `audiocapturekit` from `https://github.com/lll-solutions/AudioCaptureKit.git` (branch: main, revision: e378e950)
- `swift-asn1` from `https://github.com/apple/swift-asn1.git` (version 1.5.1)
- `swift-crypto` from `https://github.com/apple/swift-crypto.git` (version 4.2.0)

Since it's remote and branch-pinned, **caching will absolutely help** but won't eliminate the network check entirely due to the branch pin.

### Key Design Decisions in the Fix

1. **`-clonedSourcePackagesDirPath mac/SourcePackages`** ã Directs SPM to clone packages into a known, cacheable path instead of DerivedData (which is ephemeral).

2. **Separate `-resolvePackageDependencies` step** ã Isolates network-dependent package resolution from the build. If it hangs, the step times out cleanly rather than blocking the entire build+analysis.

3. **`-disableAutomaticPackageResolution` on the build step** ã After resolution, tells xcodebuild to use only already-resolved packages. Prevents a second resolution attempt during build.

4. **Cache key uses `Package.resolved`** ã Cache invalidates only when dependencies actually change.

5. **`restore-keys: spm-${{ runner.os }}-`** ã Falls back to any previous SPM cache if Package.resolved changed, so at least some packages are pre-cloned.

### Bonus Recommendation: Pin AudioCaptureKit to a Tag

The `branch = main` pin is the single biggest contributor to resolution slowness. Every build must check if `main` has moved. If AudioCaptureKit can publish semver tags, switching from `kind = branch` to `kind = upToNextMajorVersion` would:
- Allow SPM to skip the network check when the cached version matches
- Make builds more reproducible
- Reduce the chance of a surprise breaking change in CI

This would require a change in the Xcode project settings (the `XCRemoteSwiftPackageReference` section) and is a separate task from the workflow changes.

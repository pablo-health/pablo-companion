# Epic Skill

Orchestrate an entire beads epic through a 4-phase workflow: analyze dependencies, design review, implement with HIPAA gates, and deliver a PR.

## Usage

```
/epic EPIC-ID
/epic search terms for epic title
```

## Examples

```
/epic THERAPY-z2q
/epic SOAP note improvements
/epic authentication overhaul
```

## What This Does

The orchestrator (Claude Code) acts as epic manager and coordinates a team of specialized agents through 4 phases:

1. **Analyze** — Planner maps dependency DAG into execution batches, flags HIPAA touchpoints
2. **Design Review** — Reads or creates design doc, presents for user approval
3. **Implement** — Coder + HIPAA reviewer work through batches with quality gates
4. **Deliver** — Creates PR with full traceability (design doc, tasks, HIPAA checklist)

Every phase has a **user approval gate**. No work proceeds without your sign-off.

## Instructions

When the user invokes this skill:

### Step 1: Resolve the Epic

1. Parse the input to determine if it's an epic ID or search term
2. If it looks like an ID (e.g., `THERAPY-z2q`), run `bd show <id>`
3. If it's a search term, run `bd search <term>` to find matching epics
4. If multiple matches, ask the user to select one
5. If no matches, suggest creating a new epic with `bd create`

### Step 2: Check for Resume

1. Read all child tasks of the epic via `bd show`
2. If some tasks are already closed, report: "Resuming epic — N of M tasks already completed"
3. Show which tasks remain open

### Step 3: Launch Epic Team Workflow

**CRITICAL**: Read `.claude/agents/epic-team.md` FIRST before doing anything else. That file defines the full 4-phase workflow, team structure, and agent roles. Execute it as written.

From this point forward, you are the **coordinator, not the doer**. Delegate all research and analysis to agents — do NOT perform deep codebase analysis, file reading, or scoping yourself. Your job is to:
- Create the team (`TeamCreate`)
- Spawn agents with clear prompts
- Present agent findings to the user at each gate
- Manage task status in beads

The 4 phases (all defined in `epic-team.md`):

- **Phase 1: Analyze** — Spawn **planner** (Explore type) to map execution batches, flag HIPAA touchpoints → present findings to user → **Gate 1**
- **Phase 2: Design Review** — Read/create design doc → present to user → **Gate 2**
- **Phase 3: Implement** — Create branch, spawn **coder** + **hipaa-reviewer** → work through batches → **Gate 3**
- **Phase 4: Deliver** — Push, create PR, close tasks, shut down team

### Key Behaviors

- **Orchestrator ≠ researcher**: Spawn agents for analysis. Your inline work should be limited to beads commands, team management, and gate presentations.
- **Session resilience**: If re-invoked on a partially-completed epic, skip closed tasks and resume
- **Scope discipline**: New work discovered during implementation becomes new beads tasks, not scope creep
- **HIPAA non-negotiable**: Any security concern blocks the PR
- **User is always in control**: Every gate requires explicit approval before proceeding

```swift
/// Generates a new 32-byte AES-256-GCM key, stores in Keychain, returns the key.
/// Called once on first launch; subsequent calls retrieve the existing key.
static func getOrCreateDeviceEncryptionKey() -> Data? {
    // Try to retrieve existing key
    if let existingKey = getToken(forKey: .deviceEncryptionKey) {
        return existingKey.data(using: .utf8) ?? Data(base64Encoded: existingKey)
    }
    
    // Generate new key: 32 bytes for AES-256
    var keyBytes = [UInt8](repeating: 0, count: 32)
    let status = SecRandomCopyBytes(kSecRandomDefault, keyBytes.count, &keyBytes)
    guard status == errSecSuccess else { return nil }
    
    let keyData = Data(keyBytes)
    let keyBase64 = keyData.base64EncodedString()
    saveToken(keyBase64, forKey: .deviceEncryptionKey)
    return keyData
}

/// Retrieves the device encryption key (must have been created via getOrCreateDeviceEncryptionKey).
static func deviceEncryptionKey() -> Data? {
    guard let base64String = getToken(forKey: .deviceEncryptionKey) else { return nil }
    return Data(base64Encoded: base64String)
}

/// Deletes the device encryption key (used for "Delete after upload" feature).
static func deleteDeviceEncryptionKey() {
    deleteToken(forKey: .deviceEncryptionKey)
}
```

## 2. Shared URL Validation (New File: `URLValidator.swift`)

Create a reusable validator to avoid duplication across LoginView, SettingsView, APIClient, and fetchServerConfig.

```swift
// File: mac/PabloCompanion/Services/URLValidator.swift

import Foundation

struct URLValidator {
    /// Validates that a URL string has a secure HTTPS scheme (or http://localhost in DEBUG).
    /// Returns nil for valid URLs, error message string for invalid URLs.
    static func validateScheme(_ urlString: String) -> String? {
        let trimmed = urlString.trimmingCharacters(in: .whitespaces)
        guard !trimmed.isEmpty else {
            return "URL cannot be empty"
        }
        
        guard let url = URL(string: trimmed) else {
            return "Invalid URL format"
        }
        
        #if DEBUG
        // In DEBUG: allow https:// or http://localhost
        if url.scheme == "https" {
            return nil  // Valid
        }
        if url.scheme == "http" && url.host == "localhost" {
            return nil  // Valid for local development
        }
        return "URL must use HTTPS (or http://localhost in DEBUG)"
        #else
        // In production: HTTPS only
        if url.scheme == "https" {
            return nil  // Valid
        }
        return "URL must use HTTPS"
        #endif
    }
    
    /// Ensures a URL has HTTPS scheme (or http://localhost in DEBUG).
    /// Throws if validation fails.
    static func throwIfInvalid(_ urlString: String) throws {
        if let error = validateScheme(urlString) {
            throw URLValidationError.invalidScheme(error)
        }
    }
}

enum URLValidationError: LocalizedError {
    case invalidScheme(String)
    
    var errorDescription: String? {
        switch self {
        case let .invalidScheme(msg):
            return msg
        }
    }
}
```

```swift
// File: mac/PabloCompanion/Services/ProductionEncryptor.swift

import AudioCaptureKit
import Crypto
import Foundation

/// Production encryptor using AES-256-GCM with device key stored in Keychain.
struct ProductionEncryptor: CaptureEncryptor {
    private let key: SymmetricKey
    
    init?() {
        guard let keyData = KeychainManager.deviceEncryptionKey() else {
            // Key not found; initialize it
            guard let newKeyData = KeychainManager.getOrCreateDeviceEncryptionKey() else {
                return nil
            }
            self.key = SymmetricKey(data: newKeyData)
            return
        }
        self.key = SymmetricKey(data: keyData)
    }
    
    var algorithm: String {
        "AES-256-GCM"
    }
    
    func encrypt(_ data: Data) throws -> Data {
        do {
            let sealedBox = try AES.GCM.seal(data, using: key)
            guard let combined = sealedBox.combined else {
                throw CaptureError.encryptionFailed("Failed to produce combined sealed box")
            }
            return combined
        } catch let error as CaptureError {
            throw error
        } catch {
            throw CaptureError.encryptionFailed(error.localizedDescription)
        }
    }
    
    func decrypt(_ data: Data) throws -> Data {
        do {
            let sealedBox = try AES.GCM.SealedBox(combined: data)
            return try AES.GCM.open(sealedBox, using: key)
        } catch {
            throw CaptureError.encryptionFailed("Decryption failed: \(error.localizedDescription)")
        }
    }
    
    func keyMetadata() -> [String: String] {
        [
            "keyId": "device-key-v1",
            "algorithm": algorithm,
        ]
    }
}
```

```swift
static func decryptFileData(at url: URL) throws -> Data {
    guard let encryptor = ProductionEncryptor() else {
        throw CaptureError.encryptionFailed("Encryption key not found")
    }
    
    let fileData = try Data(contentsOf: url)
    let headerSize = 44
    guard fileData.count > headerSize else {
        throw CaptureError.encryptionFailed("File too small to contain WAV header")
    }

    let header = fileData.prefix(headerSize)
    var offset = headerSize
    var pcmData = Data()

    while offset + 4 <= fileData.count {
        let lengthBytes = fileData[offset ..< offset + 4]
        let chunkLength = lengthBytes.withUnsafeBytes { $0.load(as: UInt32.self) }
        let length = Int(UInt32(littleEndian: chunkLength))
        offset += 4

        guard offset + length <= fileData.count else {
            throw CaptureError.encryptionFailed("Encrypted chunk exceeds file bounds")
        }

        let chunkData = fileData[offset ..< offset + length]
        let decrypted = try encryptor.decrypt(Data(chunkData))
        pcmData.append(decrypted)
        offset += length
    }

    var wav = Data(header)
    let dataSize = UInt32(pcmData.count)
    let fileSize = UInt32(36 + pcmData.count)
    wav.replaceSubrange(4 ..< 8, with: withUnsafeBytes(of: fileSize.littleEndian) { Data($0) })
    wav.replaceSubrange(40 ..< 44, with: withUnsafeBytes(of: dataSize.littleEndian) { Data($0) })
    wav.append(pcmData)

    return wav
}
```

## Research Findings: PABLO-D-100 — Firebase API Key in Header vs. Query Param

### Summary

The `X-Goog-Api-Key` header **is the standard Google Cloud method** for passing API keys to REST APIs, and the production `securetoken.googleapis.com` endpoint **very likely supports it** — but Firebase's own documentation never explicitly shows it for this endpoint, and there is no confirmed production test in public sources.

### Detailed Findings

**1. Google Cloud standard (documented):**
Google's official docs at [Use API keys to access APIs](https://docs.cloud.google.com/docs/authentication/api-keys-use) say to use `X-Goog-Api-Key` header and warn that the `?key=` query param "exposes your key to theft through URL scans." Example:
```
curl -X POST \
    -H "X-goog-api-key: API_KEY" \
    -H "Content-Type: application/json" \
    "https://translation.googleapis.com/language/translate/v2"
```

However, the same page cautions: **"Not all Google Cloud APIs accept API keys to authorize usage."** You must check each API individually.

**2. Firebase Auth REST API docs (no header mention):**
The [Firebase Auth REST API docs](https://firebase.google.com/docs/reference/rest/auth) and the [Identity Platform REST API docs](https://docs.cloud.google.com/identity-platform/docs/use-rest-api) show the token refresh endpoint exclusively with the query parameter:
```
https://securetoken.googleapis.com/v1/token?key=[API_KEY]
```
Neither page mentions `X-Goog-Api-Key` as an alternative. The only non-standard header mentioned is `X-Firebase-Locale` (for localization).

**3. Firebase emulator issue (strong indirect evidence):**
[GitHub issue #5249](https://github.com/firebase/firebase-tools/issues/5249) reported that the Firebase Auth **emulator** did not accept `X-Goog-Api-Key`. The fix (PR #5263, merged Dec 2022) added header support to the emulator. The key detail: the issue author stated the header "should work" per Google Cloud docs, and the Firebase team agreed and fixed the emulator to match production behavior. This **strongly implies the production endpoint already accepts the header**.

**4. Firebase API key docs:**
The [Firebase API keys page](https://firebase.google.com/docs/projects/api-keys) does not mention `X-Goog-Api-Key` at all and focuses on how Firebase keys are "identification only, not authorization."

### Risk Assessment

| Factor | Assessment |
|--------|-----------|
| Will it work in production? | **Very likely yes** — Google Cloud convention, emulator was patched to match production |
| Is it documented for this endpoint? | **No** — Firebase docs only show query param |
| What if it fails? | Token refresh breaks → user gets logged out → they have to re-login |
| Reversibility | Easy — change one URL construction line back |

### Recommendation

**Implement PABLO-D-100, but with a safety net:**

1. The change is small (modify `TokenRefresher.swift:43` to drop `?key=` from the URL and add `X-Goog-Api-Key` header to the URLRequest).
2. **Test against production Firebase** before merging. This is the only way to confirm it works — no emulator test is sufficient.
3. Keep a fallback plan: if the header approach fails in production testing, close D-100 as "won't-do — Firebase endpoint only documents query param."

The security benefit is real but modest: the Firebase Web API key is already semi-public by design (embedded in client apps, not a secret), so the exposure via server logs is a low-severity issue. This is a "defense in depth" improvement, not a critical fix.

### Alternative Considered

If the header doesn't work, there's no other way to avoid the key in the URL for this specific endpoint. Firebase's API design requires it. The only alternative would be proxying token refresh through your own backend, which adds latency and complexity for minimal security gain (since the key isn't truly secret).

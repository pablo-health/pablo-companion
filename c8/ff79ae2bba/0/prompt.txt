## Task 1: Extract RecordingService (PABLO-D-b2q)

**Problem:** `RecordingViewModel.swift` is 550 lines â€” nearly double the 300-line limit. All AudioCaptureKit orchestration lives in the ViewModel, which mixes UI state with recording lifecycle logic.

**Approach:** Extract a `@MainActor` `RecordingService` class that owns the AudioCaptureKit session, then have `RecordingViewModel` delegate to it.

### What moves to RecordingService
- `CompositeCaptureSession` instance ownership
- `startRecording()`, `pauseRecording()`, `resumeRecording()`, `stopRecording()` core logic
- Permission checks (`requestMicrophonePermission`, `CGPreflightScreenCaptureAccess`)
- Device enumeration (`availableAudioSources()`)
- CoreAudio device-change listener
- File writing / `LocalRecording` construction

### What stays in RecordingViewModel
- `@Published` / `@Observable` state properties (for SwiftUI binding)
- UI-facing computed properties (recording duration display, level meters)
- Delegation from `RecordingService` back to update published state
- Test tone generation (UI-triggered smoke test)
- Playback logic (separate concern, stays in ViewModel or extracted later)

### File
- **New:** `mac/PabloCompanion/Services/RecordingService.swift`
- **Modified:** `mac/PabloCompanion/ViewModels/RecordingViewModel.swift` (reduce from ~550 â†’ ~200 lines)

### Interface shape
```swift
@MainActor
final class RecordingService: ObservableObject {
    func startRecording(sessionId: UUID) async throws -> URL
    func pauseRecording()
    func resumeRecording()
    func stopRecording() async -> LocalRecording?
    func requestPermissions() async -> Bool
    func availableAudioSources() async -> [AudioSource]
}
```

Here’s the practical reality for HIPAA-grade mental health transcription with diarization:
If “on-device only” is a hard requirement, you’ll generally get the best results by running an offline ASR model (e.g., Whisper) plus a local diarization pipeline.
Native OS speech APIs on macOS/Windows can work, but they’re not a slam dunk for “always offline + consistent diarization + clinical accuracy,” and you’ll want to verify exactly when they hit the network.
Below is a solid approach, plus the algorithms for “voice sample → person label” (therapist/patient/couple).
1) HIPAA constraints that matter for architecture
HIPAA compliance isn’t just “no cloud.” It’s mainly: access controls, auditability, encryption at rest/in transit, retention controls, least-privilege, breach procedures, etc. But cloud transcription usually triggers a BAA requirement (if the vendor touches PHI). The simplest way to reduce compliance scope is: process locally, store locally, and don’t transmit audio off-device.
So the decision tree is basically:

Strict on-device only → offline ASR + offline diarization + local storage
Cloud allowed with BAA → use a vendor that will sign a BAA (often easier operationally, but not your constraint)
2) Are macOS + Windows native speech libraries “good enough” offline?
macOS / Apple Speech
Apple’s Speech framework is the official API for speech recognition on Apple platforms.
However, “Speech framework” ≠ “guaranteed offline in every configuration.” Apple has been moving more intelligence on-device (and they publicly emphasize on-device privacy for some Apple Intelligence capabilities), but you should treat “offline always” as something you prove in your own environment (firewall the process, watch network calls, verify behavior per OS version / locale / model). 
Big limitation for your use case: Apple’s native stack does not give you a turnkey, production-grade multi-party diarization pipeline with speaker enrollment + labeling. You’ll still end up building diarization yourself.

Windows
Windows has built-in dictation/voice features, but “offline and developer-controllable” varies widely by API and configuration. For truly offline on-device STT in Microsoft-land, there are offerings explicitly designed for that (example: Embedded Speech under Azure AI Speech for on-device scenarios). 
That said, this is a separate product path with its own packaging/licensing and is not the same as “whatever ships in the OS.”
Bottom line:

Native OS libraries can be part of a solution, but for consistent offline behavior + diarization + clinical customization, most teams end up using a dedicated offline ASR model + diarization components they fully control.
3) A strong on-device stack (cross-platform) that’s common in practice
A) ASR (speech-to-text)
Whisper-family models are the usual starting point for high accuracy and broad language robustness, and can run fully offline via ports like whisper.cpp. (Whisper itself doesn’t do diarization.) 
Practical picks:

whisper.cpp (C/C++): easiest to embed, very portable
faster-whisper / CTranslate2 (Python): often faster on desktop CPUs/GPUs
B) Diarization (“who spoke when”)
Typical modern diarization pipeline:
VAD (voice activity detection) to find speech regions
Speaker embedding model to produce vectors per segment (x-vectors / ECAPA / etc.)
Clustering (AHC, spectral clustering) to group segments by speaker
Re-segmentation / smoothing (optional)
Overlap handling (optional but important in couples)
Toolkits:
pyannote.audio is widely used and MIT-licensed as a toolkit. 
(You’ll still want to confirm the license/terms for any specific pretrained weights you ship, and how you download/cache them for offline use.)
NVIDIA NeMo / SpeechBrain also have diarization components (not pulled here, but they’re common alternatives).
C) Merge diarization + transcript
You can run diarization and ASR separately, then align: “speaker segment time window → assign speaker label to words/tokens in that window.” This is a common pattern (and is exactly how many “Whisper + diarization” recipes work). 
4) “2 people is easy” — yes, and you can do better than generic Speaker 1 / Speaker 2
If it’s therapist + patient:
Run diarization → you get clusters A/B.
Decide which cluster is the therapist.
How to map cluster → role:
Enrollment (voice sample) for therapist (recommended): record 15–60 seconds of therapist speech during onboarding.
Compute a speaker embedding for that sample.
Compare it to each cluster’s centroid embedding (cosine similarity / PLDA). Highest match = therapist.
Label the other cluster “Patient”.
This gives you robust role labeling without manual tagging.
5) Couples counseling: associate multiple voice samples to people
Yes—this is a standard speaker recognition / speaker identification setup layered on diarization.
Option 1 (best UX + accuracy): Known-speaker diarization
You enroll each participant (Therapist, Partner A, Partner B). Then:
Perform diarization segmentation (or use VAD to segment speech)
For each segment, compute embedding
Classify segment to closest enrolled speaker (with thresholds + “unknown” fallback)
Apply temporal smoothing (HMM/Viterbi or simple median filtering)
Algorithms used:
Embeddings: ECAPA-TDNN / x-vector style networks
Similarity: cosine similarity, PLDA scoring
Decision: thresholding + smoothing
If two people talk over each other: add overlap detection and allow multi-label segments
Option 2 (no enrollment): diarize + lightweight human-in-the-loop
Diarize into 3 clusters.
Show 2–3 short sample snippets per cluster.
User assigns names once (“This is Alex, this is Jordan, this is Dr. Lee”).
Cache those cluster centroids for future sessions (with caution—voices drift with mic/room).
Option 3 (hybrid): enroll therapist only
Therapist is known; the two patients are unknown.
Label therapist via enrollment match.
Remaining clusters are “Patient 1 / Patient 2” (or ask user to name them once per session).
6) Overlap is the real “couples counseling is harder” problem
Couples sessions often have:
interruptions
laughter / backchannels
overlapping speech
Classic diarization degrades sharply here. Two practical add-ons:
Overlap detection (flag time spans likely containing >1 speaker)
Separation-aware diarization (more complex; can involve speech separation models)
If you don’t need perfect overlap attribution, a pragmatic approach is:
mark overlaps explicitly: [OVERLAP] segments
assign the dominant speaker but keep a tag that overlap occurred

# On-Device Speech Transcription & Speaker Diarization: Research Findings

## 1. Whisper Variants for Local/Embedded Use

### 1.1 whisper.cpp

**Current status**: Actively maintained under [ggml-org/whisper.cpp](https://github.com/ggml-org/whisper.cpp). Latest releases throughout 2025 with "massive performance improvements for the Metal backend, especially for beams > 1 and for quantized models." 744 public benchmark results on OpenBenchmarking.org as of Feb 2026.

**Model sizes (GGML F16 format)**:

| Model | Parameters | GGML Disk Size | VRAM Required |
|-------|-----------|---------------|---------------|
| tiny | 39M | 75 MB | ~1 GB |
| base | 74M | 142 MB | ~1 GB |
| small | 244M | 466 MB | ~2 GB |
| medium | 769M | 1.53 GB | ~5 GB |
| large-v3 | 1.55B | 3.1 GB | ~10 GB |
| **large-v3-turbo** | **809M** | **1.62 GB** | **~6 GB** |

**Quantization**: Supports Q4_0, Q4_1, Q5_0, Q5_1, Q8_0. Quantization reduces latency by ~19% and model size by ~45% while preserving accuracy. Q5_0 large-v3-turbo is roughly 1.0 GB on disk.

**Platform support**:
- **Metal (Apple Silicon)**: Full GPU inference via Metal. With CoreML + Metal, achieves 8-12x faster than CPU-only. Unified memory model means seamless weight sharing between CPU/GPU.
- **CUDA (NVIDIA)**: Full support for discrete GPUs.
- **CPU**: ARM NEON, x86 AVX, OpenVINO (Intel GPUs + CPUs).
- **CoreML/ANE**: Encoder inference on Apple Neural Engine, 3x+ speedup. First run is slow (ANE compilation), subsequent runs fast.

**Realistic processing speeds on Apple Silicon**:

| Scenario | Speed |
|----------|-------|
| Medium model, any Apple Silicon Mac | ~2-3 min to transcribe 10 min of audio (RTF ~0.2-0.3x) |
| Large-v3-turbo on M2 MacBook | ~63 seconds for 10 min of audio (RTF ~0.1x) |
| Large-v3-turbo-q5_0 with CoreML, Apple Silicon | ~1.23 seconds per short clip benchmark |
| **Extrapolated: 1 hour of audio, large-v3-turbo, M1+** | **~6-10 minutes** |

**Intel CPU comparison**: An Intel i9-9900K was "by far the slowest" in comparative benchmarks. Apple Silicon benefits massively from the Neural Engine and Metal. Intel can partially close the gap with OpenVINO acceleration.

**Key advantage**: Memory profile stays flat regardless of audio length -- critical for long therapy sessions.

### 1.2 faster-whisper / CTranslate2

**Status**: [faster-whisper](https://github.com/SYSTRAN/faster-whisper) remains the leading Python-based local transcription solution as of 2025. Up to 4x faster than vanilla Whisper with int8 quantization on both CPU and GPU. Uses CTranslate2 inference engine.

**Desktop viability**: Yes, but it is a Python library. Multiple desktop apps already embed it:
- **aTrain** (Windows Store app for transcription + diarization)
- **WhisperLive** (real-time transcription)
- **whisper-standalone-win** (standalone CLI executables for Windows/Linux/macOS)

**Platform support**: x86-64 and AArch64/ARM64 CPUs. Multiple backends: Intel MKL, oneDNN, OpenBLAS, Ruy. GPU via CUDA.

**Embedding concern for Pablo**: faster-whisper requires Python runtime. For a native desktop app with Rust core, **whisper.cpp via whisper-rs is the better choice** -- no Python dependency, direct C/Rust integration, comparable performance on Apple Silicon with Metal/CoreML.

### 1.3 Other Notable Alternatives

- **MLX Whisper** (Apple only): Runs on Apple's MLX framework. Benchmarked at 1.02s for large-v3-turbo on Apple Silicon -- slightly faster than whisper.cpp. macOS-only, not suitable for cross-platform.
- **Distil-Whisper**: 6x faster than large-v3, within 1% WER on out-of-distribution audio. 756M parameters distilled from 1.54B. Good option if fine-tuning for therapy domain.
- **Whisper Large-v3 Turbo**: The sweet spot for Pablo. 809M params, ~1.6 GB, near-large-v3 accuracy, 8x faster. **Limitation: not trained for translation tasks**, but irrelevant for English therapy sessions.
- **Parakeet (NVIDIA)**: Fastest in Apple Silicon benchmarks via CoreML bridge (0.19s), but tied to NVIDIA ecosystem.

### 1.4 Recommendation for Pablo

**whisper.cpp with large-v3-turbo (Q5_0 quantized) via whisper-rs** is the recommended path. Reasons:
- ~1 GB on disk
- ~6 min to transcribe a 1-hour therapy session on any Apple Silicon Mac
- No Python dependency
- Metal + CoreML acceleration on macOS
- CPU fallback on Windows (with CUDA option for NVIDIA GPUs)
- Actively maintained Rust bindings

## 2. Speaker Diarization for Local/Offline Use

### 2.1 pyannote.audio

**Current version**: [4.0.4](https://pypi.org/project/pyannote-audio/) (released Feb 7, 2026).

**Key models**:

| Model | License | Gated? | Quality |
|-------|---------|--------|---------|
| **community-1** | CC-BY-4.0 | **No** (just needs HF token) | Best open-source; significantly outperforms 3.1 |
| speaker-diarization-3.1 | MIT | **Yes** (must accept terms, provide company info) | Legacy; worse than community-1 |
| precision-2 | Commercial | N/A | 28% more accurate than community-1; 2x faster |

**Community-1 is the clear winner** for Pablo: CC-BY-4.0 license, no gating friction, best open-source accuracy, and a new "exclusive speaker diarization" mode that assigns exactly one speaker per frame (simplifies transcript alignment).

**Offline use**: Fully supported. Clone the model repo with git-lfs, then load from disk: `Pipeline.from_pretrained('/path/to/directory/')`. No internet needed after initial download.

**Hardware requirements**:
- Model occupies ~1-2 GB VRAM when loaded on GPU
- 6-8 GB VRAM is adequate
- CPU inference works but is slow (~700 seconds for 14 min on CPU vs. ~70 seconds on GPU for older versions)
- GPU provides ~9-10x speedup over CPU

**Input format**: Mono 16kHz. Stereo is auto-downmixed. Requires Python 3.10+ and ffmpeg.

**Limitation for Pablo**: pyannote is Python/PyTorch. Cannot be directly called from Rust. Options: (a) sidecar Python process, (b) ONNX export + Rust inference, (c) PyO3 bridge.

### 2.2 NVIDIA NeMo Diarization

**Approach**: Two systems -- cascaded pipeline (MarbleNet VAD + TitaNet embeddings + MSDD neural diarizer) and end-to-end Sortformer.

**Offline**: Yes, via `ClusteringDiarizer` class with pretrained checkpoints from NGC.

**Streaming Sortformer**: Production-grade, real-time, 2-4+ speaker tracking with minimal latency. Available on [Hugging Face](https://huggingface.co/nvidia/diar_streaming_sortformer_4spk-v2).

**Pros**: GPU-optimized, streaming capable, strong accuracy.
**Cons**: Heavy PyTorch/NeMo dependency, primarily designed for NVIDIA GPUs, complex setup, overkill for 2-speaker therapy sessions.

### 2.3 SpeechBrain Diarization

**Status**: PyTorch-based, supports ECAPA-TDNN and x-vector embeddings. Python 3.7+ on Linux/macOS only.

**Assessment**: Less mature than pyannote for diarization specifically. Better as a source of speaker embedding models (ECAPA-TDNN) than as a complete diarization pipeline.

### 2.4 C++/Rust Native Diarization

**There are no mature C++ or Rust native speaker diarization libraries as of 2025-2026.** The ecosystem is entirely Python/PyTorch. Options:

- **Kaldi** (C++): Speech recognition framework with diarization recipes, but complex and largely legacy.
- **Picovoice Falcon**: Commercial on-device diarization SDK. Cross-platform native binaries. Free tier: 250 min/month. Paid: starts at $6K/year. **Requires internet for license validation** even though processing is local. Not fully offline.
- **ONNX export**: Export pyannote/SpeechBrain models to ONNX, run via `ort` crate in Rust. Most viable path for Rust-native inference.

### 2.5 WhisperX and Combined Tools

[WhisperX](https://github.com/m-bain/whisperX) combines faster-whisper + wav2vec2 alignment + pyannote diarization in one pipeline. 70x realtime with large-v2, word-level timestamps, speaker labels. Excellent for batch processing but is a Python pipeline.

[whisper-diarization](https://github.com/MahmoudAshraf97/whisper-diarization) combines Whisper + VAD + NeMo speaker embeddings. Also Python.

A [Picovoice blog post](https://picovoice.ai/blog/whisper-cpp-speaker-diarization/) demonstrates integrating whisper.cpp + Falcon for C++ speaker diarization -- this is the closest to a Rust-compatible pipeline.

### 2.6 Recommendation for Pablo

**For a 2-speaker therapy session, the 2-channel recording approach (Section 4) largely eliminates the need for diarization.** If diarization is still needed:

- **Phase 1**: Use pyannote community-1 as a sidecar Python process, communicating with Rust core via IPC (stdin/stdout JSON, Unix socket, or gRPC).
- **Phase 2**: Export the pyannote segmentation model to ONNX and run it natively in Rust via the `ort` crate, eliminating the Python dependency.

## 3. Speaker Enrollment / Identification (Voice Biometrics)

### 3.1 Best Pretrained Model: ECAPA-TDNN

The **[SpeechBrain `spkrec-ecapa-voxceleb`](https://huggingface.co/speechbrain/spkrec-ecapa-voxceleb)** model is the go-to choice:

| Property | Value |
|----------|-------|
| Architecture | ECAPA-TDNN (Emphasized Channel Attention, Propagation and Aggregation) |
| Training data | VoxCeleb 1 + VoxCeleb 2 |
| Embedding dimension | **192** (default) or 512 |
| Pooling | Attentive statistical pooling |
| Loss | Additive Angular Margin Softmax (AAM-Softmax) |
| EER | 0.86% on Vox1_O (TaoRuijie implementation) |
| Inference time | ~69 ms per utterance |

**Alternative**: [WeSpeaker `ecapa-tdnn-vox2`](https://huggingface.co/Wespeaker/wespeaker-ecapa-tdnn512-LM) -- trained only on VoxCeleb2, provides ONNX exports. [3D-Speaker (ModelScope)](https://github.com/modelscope/3D-Speaker) released ONNX Runtime scripts for inference in April 2024.

### 3.2 Enrollment Requirements

| Parameter | Recommended |
|-----------|------------|
| Speech duration for enrollment | **5-15 seconds** of clear speech (shorter possible due to attentive pooling) |
| Embedding dimension | 192 floats = **768 bytes** per enrollment |
| Storage | Negligible -- a few KB per enrolled speaker |
| Multiple enrollment samples | Recommended -- average 3-5 embeddings for robustness |

### 3.3 Scoring: Cosine Similarity vs. PLDA

**Current best practice: Cosine similarity is sufficient and recommended for large-margin trained embeddings** (which ECAPA-TDNN is).

Research findings ([Wang et al., Interspeech 2022](https://www.isca-archive.org/interspeech_2022/wang22r_interspeech.pdf)):
- For 192-d embeddings, cosine and PLDA perform similarly
- For 512-d embeddings, PLDA degrades more than cosine
- PLDA adds complexity (requires training on held-out data, LDA dimension reduction) with minimal benefit
- An attention back-end outperforms both, but adds model complexity

**For Pablo**: Use cosine similarity. It is simple, effective, requires no additional training data, and runs in microseconds on any CPU. A threshold of ~0.25-0.35 typically works well for speaker verification.

### 3.4 Fully Offline on Laptop?

**Yes.** ECAPA-TDNN inference is ~69 ms per utterance on CPU. The model is small (under 100 MB for ONNX). No GPU required. Enrollment embeddings are stored locally (768 bytes each). Cosine similarity is trivial computation.

### 3.5 Practical Flow for Pablo

1. **Enrollment (once)**: Therapist records ~10 seconds of speech during onboarding. Extract 192-d ECAPA-TDNN embedding. Store encrypted on device.
2. **Session**: After recording, extract embeddings from short segments. Compare against enrolled therapist embedding via cosine similarity.
3. **Label**: Segments matching the therapist above threshold = "Therapist". Everything else = "Client".
4. This is far simpler and more reliable than full diarization for a known 2-party conversation.

## 4. 2-Channel Recording Approach

### 4.1 macOS: Mic + System Audio as Separate Channels

**Recommended approach (2025)**:

1. **ScreenCaptureKit** (`SCStream`) for system audio capture -- captures all audio playing through the system, available since macOS 13 (Ventura).
2. **Core Audio `CATapDescription` + `AudioHardwareCreateAggregateDevice`** to create an aggregate device combining:
   - Channel 1: Microphone input (therapist)
   - Channel 2: System audio tap (client from Zoom/Teams/Meet)
3. Process the `AudioBufferList` where channels are inherently separated.

**Alternative (simpler but less control)**: BlackHole virtual audio device + Aggregate Device in Audio MIDI Setup. However, this requires user setup and may resample audio.

**AudioCaptureKit from `audiotake2`**: Already has Pablo branding and handles macOS audio capture. This should be extended to support the 2-channel aggregate device approach.

**Known issues**:
- `AudioDeviceStart` can fail intermittently with aggregate devices
- Virtual device routing can cause resampling to 44.1 kHz
- Some apps (Slack, VLC) only play back one audio track from multi-track files

### 4.2 Windows: WASAPI Loopback + Mic

**Approach**: Run two simultaneous capture streams:

1. **WASAPI standard capture** for microphone (therapist)
2. **WASAPI loopback capture** for system audio (client from video call)
3. Mux into a 2-channel file or process independently

**Key details**:
- WASAPI loopback is fully digital -- no analog round-trip, perfect reproduction
- Works regardless of whether hardware has a loopback device
- Captures ALL system audio by default (all processes, all sessions)
- Process-specific loopback available via `ActivateAudioInterfaceAsync` with new initialization structure (Windows 10+) -- can capture only the Zoom/Teams process

**NAudio (C#/.NET)**: `WasapiLoopbackCapture` class handles this. Gotcha: `DataAvailable` does not fire during silence -- play silent audio as a workaround.

**For WinUI 3 / C#**: NAudio is the standard library. Two `WasapiCapture` instances (one standard, one loopback) writing to separate channels of a single WAV file.

### 4.3 Does 2-Channel Recording Eliminate Diarization?

**For typical therapy sessions: largely yes.** If you have:
- Channel 1 = therapist's mic (their voice, possibly some bleed from speakers)
- Channel 2 = system audio (client's voice from video call)

Then you already know "who spoke when" by which channel has energy. Simple energy-based VAD on each channel is sufficient. No speaker embedding, no clustering, no diarization model needed.

**Edge cases where diarization is still useful**:
- In-person sessions with a single mic (both speakers on one channel)
- Significant acoustic bleed between channels
- More than 2 speakers (group therapy)

**Recommendation**: Build the 2-channel recording pipeline first. Add diarization as a fallback for single-mic / in-person sessions later.

## 6. HIPAA Considerations for On-Device ML

### 6.1 Model Files and PHI

**Model files (Whisper, ECAPA-TDNN, pyannote) do NOT contain PHI.** They are trained on public datasets (LibriSpeech, VoxCeleb, etc.). They are safe to distribute with the application.

**However**: If you fine-tune models on therapy audio, the fine-tuned weights could theoretically memorize sensitive data. Do not fine-tune on PHI without de-identification.

### 6.2 Temporary Files During Processing

As of the [December 2025 HIPAA update](https://www.hipaajournal.com/hipaa-encryption-requirements/), **all ePHI must be encrypted -- no exceptions.** The previous "addressable" standard has been eliminated.

**Temporary files to secure**:
- Raw audio buffers during recording
- Intermediate audio files (WAV chunks fed to Whisper)
- Transcription output before final storage
- Speaker embedding vectors (if they can be linked to a patient)
- Any file in the system temp directory

**Requirements**:
- Encrypt all temp files with AES-256-GCM
- Use unique, rotating keys per file
- Securely delete (overwrite + unlink) temp files immediately after use
- Never write unencrypted audio to disk -- process in memory where possible
- Use platform-native secure temp directories (macOS: app sandbox container; Windows: `%LOCALAPPDATA%\Temp` within protected user profile)

### 6.3 Encryption at Rest

| Data | Encryption Requirement |
|------|----------------------|
| Audio recordings | AES-256-GCM, per-file key |
| Transcripts | AES-256-GCM |
| Speaker enrollment embeddings | AES-256 (linked to patient identity) |
| Sync queue / metadata | AES-256 if contains session details |
| Model files | Not required (no PHI) |

**Key management**:
- Store encryption keys in platform keychain (macOS Keychain, Windows Credential Manager)
- Consider envelope encryption: data encrypted with DEK, DEK encrypted with KEK stored in keychain
- Key rotation on a defined schedule
- FIPS-validated crypto libraries (Apple's CryptoKit is FIPS-validated; on Windows, use BCrypt or similar)

### 6.4 Additional Safeguards

- **Audit logging**: Log all access to audio and transcripts (who, when, what action)
- **Secure deletion**: Implement verified secure deletion with overwrite
- **In-transit**: TLS 1.3 for all sync to Pablo backend
- **No BAA needed for model providers**: Whisper/pyannote models are open-source, downloaded once, run locally. No data leaves the device, so no BAA required for model providers.
- **BAA with Pablo backend**: Required, as transcripts sync to the FastAPI backend.
